Security & Safety views — scan history, reports, findings

Interactive mockups for each open decision. Everything below is clickable where it matters — open the history popover, pin a past scan, open a finding's sheet, filter the lists. Data is realistic mock data; both views share the same patterns, shown here on Safety Scan (the richer case) with Security notes where they differ.

1 · Scan history — access & viewing model

Where history lives, and how a past scan is viewed

The core question: is the view "the latest state", or "a selected scan"? Try all three. In every model, viewing a past scan shows its findings, not just the report — and the banner/selection makes it unambiguous which scan you're looking at.

2 · Scan titles

What a scan is called in a list

The same four runs, rendered under each naming scheme. "since Jul 22, 2026" today is the scope — a property, not an identity. Security scans would read "Explicit scan · July 22" / "Automatic check · July 20" since their scope is always the whole backup.

3 · History rows — content, sort & filter

What each row shows, and how you narrow the list

Row anatomy: run date (identity) · scope + sources (what was covered) · status · severity-split findings badge · receipt (model tier / feed date). Filters and sort reuse the app's standard chip controls — try them.
